## 1. 버스 설명

Tesla 내부 CAN 버스 명명 규칙:

| Tesla 버스 이름 | 설명 | 접근 지점 |
|----------------|------|-----------|
| **ETH** | 고대역폭 CAN-FD 백본 (실제 이더넷 아님) | OBD-II 포트, X179 커넥터 |
| **CH** | 섀시 CAN — EPAS, EPB, ESP, 제동 시스템 | X179 핀 2/3 (CAN bus 4) |
| **PT** | 파워트레인 CAN — DI, BMS, 충전기 | — |
| **BDY** | 차체 CAN — VCFRONT, VCLEFT, VCRIGHT, 도어/시트 | — |
| **OBDII** | 표준 OBD-II 진단 | OBD-II 포트 |

본 프로젝트는 OBD-II / X179를 통해 **ETH** 버스에 접근합니다. FSD 관련 모든 제어 프레임은 이 버스에 존재합니다.
Nag Killer가 사용하는 EPAS 프레임(0x370)은 **CH** 버스에 있습니다.

## 3. 0x3FD (1021) `UI_autopilotControl` 프레임 상세

이것은 FSD의 가장 핵심적인 프레임입니다. **멀티플렉스** 인코딩을 사용하며, byte 0의 bit 0-2가 mux ID입니다.
펌웨어 추출 결과 이 프레임에는 **74개의 신호**가 포함되어 있습니다.

### 3.1 Mux 0 — FSD 활성화 + 상태 읽기

#### 경험적으로 검증된 비트 위치

| 기능 | 비트 위치 | 바이트/비트 | 너비 | HW3 | HW4 | 레거시 | 신호 이름 (추정) |
|------|---------|---------|------|-----|-----|--------|--------------|
| Mux ID | bit 0-2 | `data[0] & 0x07` | 3 bit | 읽기 | 읽기 | 읽기 | — |
| FSD UI 선택 감지 | bit 54 | `(data[4] >> 6) & 0x01` | 1 bit | 읽기 | 읽기 | 읽기 | `UI_fsdStopsControlEnabled` |
| **FSD 활성화** | **bit 46** | `data[5] bit 6` | 1 bit | **1 쓰기** | **1 쓰기** | **1 쓰기** | `UI_enableFullSelfDriving` |
| **FSD V14 2차 잠금** | **bit 60** | `data[7] bit 4` | 1 bit | — | **1 쓰기** | — | (HW4 전용) |
| 긴급 차량 감지 | bit 59 | `data[7] bit 3` | 1 bit | — | 선택적 1 쓰기 | — | `UI_enableApproachingEmergencyVehicleDetection` |
| **속도 오프셋 읽기** | bit 25-30 | `(data[3] >> 1) & 0x3F` | 6 bit | **읽기** | **미구현** | — | `UI_smartSetSpeedOffset` |
| **속도 모드 쓰기** | bit 49-50 | `data[6] bit 1-2` | 2 bit | **쓰기** | — | **쓰기** | `UI_autopilotDrivingProfile` |

#### 속도 오프셋 읽기 디코딩 (HW3)

```

원본 위치: byte 3, bit 1-6 (프레임 내 bit 25-30)
추출 방법: raw = (data[3] >> 1) & 0x3F
물리 값 계산: offset = clamp((raw - 30) * 5, 0, 100)
의미: 0 = 오프셋 없음, 100 = 최대 오프셋

```

코드 예시 (C++):
```cpp
speedOffset = std::max(std::min(((uint8_t)((frame.data[3] >> 1) & 0x3F) - 30) * 5, 100), 0);
```

속도 모드 쓰기 디코딩 (HW3/레거시)

```
위치: byte 6, bit 1-2
쓰기 방법: data[6] = (data[6] & ~0x06) | ((profile & 0x03) << 1)
범위: 0 = Chill, 1 = Normal, 2 = Max (Hurry)
```

3.2 Mux 1 — Nag 억제 (ECE R79)

기능 비트 위치 바이트/비트 HW3 HW4 레거시 신호 이름 (추정)
ECE R79 Nag 억제 bit 19 data[2] bit 3 0 쓰기 0 쓰기 0 쓰기 UI_applyEceR79
Enhanced Autopilot 확인 bit 46 data[5] bit 6 1 쓰기 — — —
HW4 Nag 확인 bit 47 data[5] bit 7 — 1 쓰기 — —

3.3 Mux 2 — 속도 오프셋 주입 / 속도 모드 주입

HW3: 속도 오프셋 주입

```
위치: byte 0 bit 6-7 + byte 1 bit 0-5 (프레임 내 bit 6-13, 두 바이트에 걸쳐 총 8 bit)
쓰기 방법:
  data[0] = (data[0] & ~0xC0) | ((speedOffset & 0x03) << 6)   // 하위 2 bit
  data[1] = (data[1] & ~0x3F) | (speedOffset >> 2)             // 상위 6 bit
범위: 0-100
조건: FSDEnabled == true인 경우에만 주입
```

코드 예시 (C++):

```cpp
frame.data[0] &= ~(0b11000000);
frame.data[1] &= ~(0b00111111);
frame.data[0] |= (speedOffset & 0x03) << 6;
frame.data[1] |= (speedOffset >> 2);
```

HW4: 속도 모드 주입

```
위치: byte 7 bit 4-6 (프레임 내 bit 60-62)
쓰기 방법:
  data[7] = (data[7] & ~(0x07 << 4)) | ((speedProfile & 0x07) << 4)
범위: 0 = Chill, 1 = Normal, 2 = Hurry, 3 = Max, 4 = Sloth
조건: 무조건 주입 (FSDEnabled 불필요)
```

참고: tuncasoftbildik의 구현은 << 5(bit 61-63)를 사용하며, 우리 프로젝트와 flipper 프로젝트는 << 4(bit 60-62)를 사용합니다. 한 비트 차이가 있어 경험적 검증이 필요합니다.

코드 예시 (C++):

```cpp
frame.data[7] &= ~(0x07 << 4);
frame.data[7] |= (speedProfile & 0x07) << 4;
```

HW4 Mux 2에서 펌웨어 추출로 알려진 기타 신호

신호 인덱스 신호 이름 비트 위치 상태
#68 UI_enableApproachingEmergencyVehicleDetection 알 수 없음 —
#69 UI_enableStartFsdFromParkBrakeConfirmation 알 수 없음 —
#70 UI_enableStartFsdFromPark 알 수 없음 —
#71 UI_fsdMaxSpeedOffsetPercentage 알 수 없음 역공학 필요
#72 UI_coldStartMonarchInFactory 알 수 없음 —
#73 UI_autopilotControlMux2Valid 알 수 없음 —

UI_fsdMaxSpeedOffsetPercentage는 HW4 mux 2에서 속도 오프셋과 관련된 유일한 신호이지만, 현재 비트 위치가 공개된 DBC 정의가 없습니다. CAN 패킷 캡처를 통한 역공학이 필요합니다.

4. 0x3EE (1006) DAS_autopilot 레거시 프레임 상세

레거시(HW1/HW2)는 CAN ID 1006을 사용하여 1021을 대체하며, 프레임 구조는 유사합니다:

기능 비트/바이트 설명
Mux ID data[0] & 0x07 0x3FD와 동일
FSD UI 선택 (data[4] >> 6) & 0x01 0x3FD와 동일
FSD 활성화 (mux 0) bit 46 1 쓰기
속도 모드 (mux 0) byte 6 bit 1-2 HW3와 동일 (0-2)
Nag 억제 (mux 1) bit 19 0 쓰기

레거시 프레임에는 mux 2 속도 오프셋 주입 기능이 없습니다.

5. 속도 제한 인식 관련 신호 (읽기 전용)

5.1 DAS_fusedSpeedLimit — 융합 속도 제한 (가장 정확함, 권장)

```
CAN ID: 0x399 (921) — DAS_status
버스: ETH (Chassis Bus), Party Bus에서는 0x39B (923)
시작 비트: 8
너비: 5 bit
바이트 순서: 리틀 엔디안
배율: 5
오프셋: 0
물리 범위: 0 ~ 150 kph/mph
단위: kph 또는 mph (차량 지역 설정에 따름)
주기: 500ms
특수 값: raw 0 = UNKNOWN_SNA, raw 31 = NONE
```

디코딩 방법:

```cpp
uint8_t raw = (data[1]) & 0x1F;
float speed_limit = raw * 5.0f;  // kph/mph
```

sunnypilot 실측 결과, 이 신호가 Tesla 화면에 표시되는 속도 제한과 완전히 일치함을 확인했습니다.

5.2 DAS_visionOnlySpeedLimit — 비전 전용 인식 속도 제한

오프셋: 0
물리 범위: 0 ~ 150 kph/mph

6. 속도 오프셋 및 속도 설정 신호

6.1 UI_userSpeedOffset — 사용자 UI 설정 속도 오프셋

이것은 사용자가 Tesla UI에서 설정한 Autopilot 속도 오프셋 값입니다(+1에서 +10 등).

```
CAN ID: 0x3D9 (985) — UI_gpsVehicleSpeed (Model 3/Y)
        0x2F8 (760) — Model S/X
시작 비트: 40
너비: 6 bit
바이트 순서: 리틀 엔디언 (@1+)
배율: 1
오프셋: -30
물리 범위: -30 ~ +33 kph/mph
단위: kph 또는 mph (UI_userSpeedOffsetUnits에 의해 결정)
```

DBC 정의:

```
SG_ UI_userSpeedOffset : 40|6@1+ (1,-30) [-30|33] "kph/mph" DAS
```

디코딩 방법:

```cpp
uint8_t raw = (data[5]) & 0x3F;
int offset = (int)raw - 30;  // 물리 값, 단위 kph 또는 mph
```

연관 신호:

```
UI_userSpeedOffsetUnits   (bit 47, 1 bit): 0=MPH, 1=KPH
UI_mapSpeedLimitUnits     (bit 46, 1 bit): 0=MPH, 1=KPH
```

6.2 UI_smartSetSpeedOffset — 0x3FD mux 0의 속도 오프셋

이것은 0x3FD 프레임 mux 0에 있는 속도 오프셋 신호로, 현재 HW3Handler에서 사용 중입니다.

```
CAN ID: 0x3FD (1021) — UI_autopilotControl, Mux 0
위치: byte 3, bit 1-6 (프레임 내 bit 25-30)
너비: 6 bit
인코딩 방식: raw = physical + 30
```

디코딩 방법 (현재 프로젝트 구현):

```cpp
int raw = (int)((frame.data[3] >> 1) & 0x3F) - 30;
int offset = clamp(raw * 5, 0, 100);
```

연관 신호 (펌웨어 추출, 공개된 비트 위치 없음):

```
UI_smartSetSpeedOffsetType: 열거형 FIXED_OFFSET=0, PERCENTAGE_OFFSET=1
UI_automaticSetSpeedOffset: 자동 속도 오프셋
UI_smartSetSpeed:           스마트 설정 속도
```

6.3 UI_fsdMaxSpeedOffsetPercentage — FSD 최대 속도 오프셋 백분율

```
CAN ID: 0x3FD (1021) — UI_autopilotControl, Mux 2
신호 인덱스: #71 (펌웨어 추출)
비트 위치: 알 수 없음 ← CAN 패킷 캡처 역공학 필요
너비: 알 수 없음
설명: FSD의 속도 상한 백분율을 제어합니다. 펌웨어에 이 신호가 존재하지만, 공개된 DBC에는 비트 위치가 정의되어 있지 않습니다.
```

6.4 DAS_accSpeedLimit — ACC/Autopilot 속도 상한

```
CAN ID: 0x389 (905) — DAS_status2

7. ISA 속도 제한 알림음 억제

CAN ID: 0x399 (921) — DAS_status

HW4만 유효합니다. byte 1의 bit 5를 설정하여 ISA 속도 제한 알림음을 억제합니다.

수정 비트:

```
byte 1, bit 5 → 1 쓰기 (data[1] |= 0x20)
```

체크섬을 다시 계산해야 합니다 (byte 7):

```
checksum = sum(byte[0]..byte[6]) + (CAN_ID & 0xFF) + (CAN_ID >> 8)
         = sum(byte[0]..byte[6]) + 0x99 + 0x03
byte[7] = checksum & 0xFF
```

코드 예시 (C++):

```cpp
frame.data[1] |= 0x20;
uint8_t sum = 0;
for (int i = 0; i < 7; i++)
    sum += frame.data[i];
sum += (921 & 0xFF) + (921 >> 8);  // 0x99 + 0x03
frame.data[7] = sum & 0xFF;
```

DAS_status 프레임의 기타 신호 (CAN 0x399)

신호 이름 비트 너비 설명
DAS_autopilotState — — 자율주행 상태
DAS_fusedSpeedLimit 8 5 bit 융합 속도 제한 (×5 kph)
DAS_suppressSpeedWarning 13 1 bit 과속 경고 억제
DAS_visionOnlySpeedLimit 16 5 bit 비전 전용 속도 제한 (×5 kph)
DAS_blindSpotRearLeft — — 좌측 후방 사각지대
DAS_blindSpotRearRight — — 우측 후방 사각지대
DAS_summonAvailable — — 스마트 호출 가능 여부

8. Nag 억제 (핸들 조작 감지)

방법 1: 0x3FD mux 1 ECE R79 비트 클리어

3.2절을 참조하십시오. bit 19 = 0으로 설정합니다 (HW4는 추가로 bit 47 = 1로 설정).

방법 2: CAN 880 (0x370) Counter+1 Echo

버스: CH (섀시 CAN), 다른 물리적 접근 지점이 필요합니다.

원리: EPAS3P_sysStatus 프레임을 수신하여, handsOnLevel == 0(핸들 미감지, nag 발생 임박)일 때 가짜 프레임을 구성합니다:

1. 원본 프레임을 복사합니다
2. handsOnLevel = 1로 설정합니다 (byte 4 bit 6-7 → 01)
3. torsionBarTorque = 0xB6 (고정 토크 1.80 Nm)
4. Counter +1 (byte 6 하위 4 bit)
5. 체크섬 재계산 (byte 7)

가짜 프레임이 실제 프레임보다 먼저 도착하면, 실제 프레임은 카운터가 동일하여 중복 프레임으로 폐기됩니다.

```
CAN ID: 0x370 (880) — EPAS3P_sysStatus
버스: CH (Chassis CAN)

읽기 필드:
  handsOnLevel: (data[4] >> 6) & 0x03  — 0=핸들 미감지, 1=약한 파지, 2/3=강한 파지
  counter:      data[6] & 0x0F

Echo 프레임 구성:
  echo.data[0] = frame.data[0]
  echo.data[1] = frame.data[1]
  echo.data[2] = (frame.data[2] & 0xF0) | 0x08   // 토크 품질 nibble
  echo.data[3] = 0xB6                              // torsionBarTorque = 1.80 Nm
  echo.data[4] = frame.data[4] | 0x40              // handsOnLevel = 1
  echo.data[5] = frame.data[5]
  echo.data[6] = (frame.data[6] & 0xF0) | ((counter + 1) & 0x0F)
  echo.data[7] = checksum

체크섬: sum(byte[0]..byte[6]) + 0x70 + 0x03
```

코드 예시 (C++):

```cpp
uint8_t cnt = (frame.data[6] & 0x0F);
cnt = (cnt + 1) & 0x0F;
echo.data[6] = (frame.data[6] & 0xF0) | cnt;

uint16_t sum = 0;
for (int i = 0; i < 7; i++)
    sum += echo.data[i];
sum += (0x370 & 0xFF) + (0x370 >> 8);  // 0x70 + 0x03
echo.data[7] = (uint8_t)(sum & 0xFF);
```

테스트 완료: Model Y Performance 2022 HW3, Basic Autopilot, X179 pin 2/3 (CAN bus 4)

20. 체크섬 계산 방법

Tesla는 통합된 체크섬 알고리즘을 사용합니다 (0x399, 0x370, 0x313 등의 프레임에 적용):

```
checksum = sum(byte[0] .. byte[N-2]) + (CAN_ID & 0xFF) + (CAN_ID >> 8)
byte[N-1] = checksum & 0xFF   // 마지막 바이트가 체크섬
```

여기서 N = DLC (보통 8), 체크섬 바이트는 일반적으로 byte[7]입니다.

코드 예시 (C++, 현재 프로젝트 computeTeslaChecksum 함수):

```cpp
uint8_t computeTeslaChecksum(const CanFrame &frame, uint8_t checksumByteIndex = 7)
{
    uint16_t sum = (uint16_t)(frame.id & 0xFF) + (uint16_t)((frame.id >> 8) & 0xFF);
    for (uint8_t i = 0; i < frame.dlc; ++i) {
        if (i == checksumByteIndex) continue;
        sum += frame.data[i];
    }
    return (uint8_t)(sum & 0xFF);
}
```

각 프레임의 CAN ID 분할:

CAN ID Hex 하위 바이트 상위 바이트 상수 합계
921 0x399 0x99 0x03 0x9C
880 0x370 0x70 0x03 0x73
787 0x313 0x13 0x03 0x16
